1. Назначение заголовка Content-Type

Content-Type указывает тип (медиатип) тела HTTP‑запроса или ответа — то есть как интерпретировать содержимое сообщения. 
Примеры: text/html; charset=utf-8, application/json, multipart/form-data; boundary=---.
 Сервер или клиент смотрит на этот заголовок чтобы правильно распарсить тело: декодировать JSON, обработать форму, сохранить файл и т.д.
 
2. Назначение заголовка Accept

Accept сообщает серверу, какие медиатипы клиент готов принять в ответе. 
Например Accept: application/json, text/html;q=0.8 означает приоритет JSON, затем HTML. 
Сервер использует это для выбора формата ответа (content negotiation) или возвращает 406 Not Acceptable, если ничего подходящего нет.


3. Для чего используется значение multipart/form-data заголовка Content-Type

multipart/form-data применяется для передачи данных формы, когда в ней есть файлы или бинарные части. 
Тело сообщения разбивается на части (части разделяются границей — boundary), каждая часть содержит собственные заголовки (например Content-Disposition, Content-Type) и данные. 
Это позволяет одновременно передать текстовые поля и один или несколько файлов в одном POST‑запросе.

4. Как с помощью тега form обеспечить значение multipart/form-data заголовка Content-Type

В HTML указывают атрибут enctype у тега form:
<form action="/upload" method="post" enctype="multipart/form-data">
При этом браузер при отправке установит заголовок Content-Type: multipart/form-data; boundary=... и упакует поля и файлы в соответствующие части.

5. Какое значение заголовка Content-Type отправляется тегом form в запросе по умолчанию

По умолчанию (если enctype не указан) для method="post" браузер использует application/x-www-form-urlencoded. 
Это кодировка, где пары имя=значение кодируются в URL‑подобном формате (с percent‑encoding и заменой пробелов на +) и помещаются в тело запроса.

6. Где и в каком формате передаются параметры в GET‑запросе?

Параметры передаются в строке запроса (query string) URL после знака ?, в форме ?name=value&foo=bar.
Значения и имена кодируются URL‑encoding (percent‑encoding), например пробел → %20 или + в некоторых контекстах. 
Тело GET‑запроса обычно пустое и параметры в нём не передают.

7.  Где и в каком формате передаются параметры в POST‑запросе?

В POST параметры обычно находятся в теле запроса. Формат зависит от Content-Type:

    application/x-www-form-urlencoded — name=value&foo=bar (URL‑кодирование);
    multipart/form-data — части с заголовками и телами (используется для файлов);
    application/json — JSON‑строка, например {"name":"Ivan","age":30}. Также возможны другие типы (XML, plain text и т.д.), всё определяется Content-Type.
    
    
8. Поясните понятие JSON

JSON (JavaScript Object Notation) — текстовый формат для представления структурированных данных: объектов (пар «ключ: значение») и массивов. 
Простой и компактный, широко используется для обмена данными между клиентом и сервером: {"name":"Anna","age":25,"tags":["js","api"]}.
 Легко парсится во многих языках; MIME‑тип обычно application/json.  
 
9. Поясните понятие XML

XML (eXtensible Markup Language) — текстовый формат для описания структурированных данных с помощью вложенных тегов и атрибутов. Пример: <person><name>Anna</name><age>25</age></person>. 
XML более вербозен, поддерживает схемы (XSD), пространства имён (namespaces) и множество инструментов для валидации и трансформации (XSLT).
 MIME‑тип обычно application/xml или text/xml.
